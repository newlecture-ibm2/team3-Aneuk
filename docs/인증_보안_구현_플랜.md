# 🚀 인증/보안(Security) 및 JWT 기반 로그인 구현 마스터 플랜

해당 문서는 Jira 티켓 AN-115(인증, security) 관련 하위 티켓들(AN-152 ~ AN-157)을 해결하기 위한 작업 순서와 설계 방향을 정리한 마스터 플랜입니다. 프로젝트 아키텍처 규칙(BFF 패턴, 헥사고널 아키텍처, HttpOnly 쿠키 정책 등)을 철저히 준수하여 5단계로 설계되었습니다.

---

## 🟢 1단계: Spring Security & JWT 기반 공사 (사전 준비)
*어떤 유저가 접근하든 공통으로 거치게 될 보안 필터와 토큰 생성기를 만듭니다.*

1. **`JwtProvider` 유틸리티 클래스 설계**
   - 위치: `backend/src/main/java/com/anook/backend/global/util`
   - 내용: 비밀키(Secret Key) 설정 및 JWT 발급/검증 로직. 토큰의 Claim(내용)에 사용자의 역할(GUEST, STAFF, ADMIN) 및 식별자(방번호, PIN, ID)를 포함하도록 구성.
2. **`JwtAuthFilter` 커스텀 필터 설계**
   - 위치: `backend/src/main/java/com/anook/backend/security`
   - 내용: 클라이언트의 요청 헤더에서 **Authorization (Bearer 토큰)**을 추출하여 검증하는 필터 작성. 검증 성공 시 `SecurityContext`에 유저 권한 정보 세팅.
3. **`SecurityConfig` 환경 설정 (기존 파일 활용)**
   - 위치: `backend/src/main/java/com/anook/backend/config/SecurityConfig.java`
   - 내용: 기존 파일에 폼 로그인 비활성화, `SessionCreationPolicy.STATELESS` (세션 미사용) 설정을 추가.
   - 엔드포인트 권한 분리:
     - `/auth/**` ➡️ `permitAll` (인증 전 접근 가능)
     - `/admin/**` ➡️ `hasRole('ADMIN')`
     - `/staff/**` ➡️ `hasRole('STAFF')`
     - `/chat/**` ➡️ `hasRole('GUEST')`
4. **`RoleHierarchy` 설정 추가**
   - 내용: `ROLE_ADMIN > ROLE_STAFF` 계층을 선언하여, ADMIN 권한자가 `/staff/**` 경로에도 접근할 수 있도록 설정.

## 🟡 2단계: 도메인 및 어댑터 설계 (AN-152, 153, 154 준비)
*인증에 필요한 핵심 도메인 모델과 DB 연동을 헥사고널 모듈 구조에 맞게 준비합니다.*

1. **도메인 모델**
   - 위치: `backend/src/main/java/com/anook/backend/admin/guest/domain/model/` 및 `admin/staff/domain/model/`
   - 내용: `GuestSession` (방 번호), `Staff` (PIN 번호, 이름, 부서-is_admin 포함) 엔티티 정의. (Admin은 Staff에 통합)
2. **Repository Port**
   - 위치: 각 모듈의 `application/port/out/`
   - 내용: `StaffRepositoryPort` (PIN으로 직원 및 소속 부서 조회) 등 인터페이스 설계.
3. **Persistence Adapter**
   - 위치: 각 모듈의 `adapter/out/persistence/`
   - 내용: 위 Port를 구현하여 실제 JPA Repository와 도메인 모델을 매핑(Mapping)하는 어댑터 작성. (Security 모듈은 인증 시 이 포트를 호출하거나, 필요시 Security 전용 어댑터를 별도로 둡니다.)

## 🟠 3단계: 역할별 인증 비즈니스 로직 및 API 구현 (AN-152, 153, 154, 155)
*로그인을 검증하고 JWT를 발급하는 실질적인 엔드포인트를 `security` 모듈에 통합하여 만듭니다.*

1. **투숙객 로직 (AN-152)**
   - Service: `GuestAuthService` (정적 QR 방번호 검증 → guest 존재 확인 → JWT 생성)
   - Controller: `GuestAuthController` ➡️ `@PostMapping("/api/auth/guest")`
   - 위치: `backend/src/main/java/com/anook/backend/security`
2. **직원/관리자 통합 로직 (AN-153, AN-154)**
   - Service: `StaffAuthService` (DB의 PIN 검증 → 소속 부서의 `is_admin` 여부에 따라 JWT에 `ADMIN` 또는 `STAFF` Role 부여)
   - Controller: `StaffAuthController` ➡️ `@PostMapping("/api/auth/staff")` (Admin 분리 없이 단일 엔드포인트 사용)
   - 위치: `backend/src/main/java/com/anook/backend/security`
3. **API 반환 공통 규칙 (AN-155)**
   - Controller에서 발급된 JWT를 직접 쿠키로 굽지 않고, **JSON 형태의 응답 본문(Body)** `{ "token": "eyJ...", "role": "..." }` 으로 반환. (Next.js 서버가 이를 받아 iron-session으로 쿠키화합니다.)

## 🔴 4단계: 동시 접속 제어 로직 (AN-156)
*보안 강화를 위한 다중 기기 세션 차단 기능입니다.*

1. **저장소 활용 (Redis 또는 DB)**
   - 직원이 PIN으로 로그인하면, 발급된 토큰 고유값(JTI)을 Redis 등에 캐싱.
2. **중복 접속 차단 (`StaffAuthService`)**
   - 동일한 PIN으로 새로운 로그인이 감지되면 기존에 저장된 토큰을 무효화(블랙리스트 처리)하여, 이전 기기에서 활동하려 할 때 필터에서 차단되도록 설계.

## 🔵 5단계: 프론트엔드 연동 및 화면 구현 (AN-157)
*스프링부트에서 완성된 인증 체계를 Next.js 브라우저 화면과 연결합니다.*

1. **BFF 라우트 설정**
   - 위치: `frontend/src/app/api/auth/[...path]/route.ts`
   - 내용: 클라이언트의 로그인 요청을 Spring Boot로 프록시. Spring Boot가 넘겨준 JWT JSON 응답을 받아 **iron-session을 활용하여 암호화된 세션 쿠키(`app_session`)**로 브라우저에 저장. 
   - 이후 API 요청 시: Next.js 프록시가 `app_session` 쿠키를 복호화해 JWT를 꺼내고, Spring Boot 요청 시 `Authorization: Bearer ...` 헤더에 담아 전송.
2. **로그인 UI 컴포넌트 개발**
   - 직원/관리자 통합 PIN 화면: `frontend/src/app/(auth)/staff/login/_components/StaffLoginForm/`
   - 규칙: 관리자 로그인 화면 분리 없이 PIN 화면 하나로 통합. JWT에서 꺼낸 Role을 기반으로 메인 화면 이동 분기 처리.
3. **전역 상태 관리**
   - 위치: `frontend/src/store/useAuthStore.ts`
   - 내용: 보안상 토큰은 브라우저에서 읽지 못하므로, BFF에서 전달받은 '사용자 권한 정보(Role)와 닉네임'만 Zustand 스토어에 올려 UI 분기 처리에 활용.
