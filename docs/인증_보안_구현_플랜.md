# 🚀 인증/보안(Security) 및 JWT 기반 로그인 구현 마스터 플랜

해당 문서는 Jira 티켓 AN-115(인증, security) 관련 하위 티켓들(AN-152 ~ AN-157)을 해결하기 위한 작업 순서와 설계 방향을 정리한 마스터 플랜입니다. 기존 아키텍처 제안 구조(`config` 및 `security` 패키지 통합)와 HttpOnly 쿠키 정책을 철저히 준수하여 5단계로 설계되었습니다.

---

## 🟢 1단계: Spring Security & JWT 기반 공사 (사전 준비)
*어떤 유저가 접근하든 공통으로 거치게 될 보안 필터와 토큰 생성기를 `security` 패키지에 통합하여 만듭니다.*

1. **`JwtProvider` 유틸리티 클래스 설계**
   - 위치: `backend/src/main/java/com/anook/backend/security`
   - 내용: 비밀키(Secret Key) 설정 및 JWT 발급/검증 로직. 토큰의 Claim(내용)에 사용자의 역할(GUEST, STAFF, ADMIN) 및 식별자(방번호, PIN, ID)를 포함하도록 구성.
2. **`JwtAuthFilter` 커스텀 필터 설계**
   - 위치: `backend/src/main/java/com/anook/backend/security`
   - 내용: 기존의 `Authorization` 헤더 방식이 아닌, 프론트엔드 규칙에 따라 **HttpOnly Cookie**에서 JWT를 추출하여 검증하는 필터 작성. 검증 성공 시 `SecurityContext`에 유저 권한 정보 세팅.
3. **`SecurityConfig` 환경 설정 (기존 파일 활용)**
   - 위치: `backend/src/main/java/com/anook/backend/config/SecurityConfig.java`
   - 내용: 기존 파일에 폼 로그인 비활성화, `SessionCreationPolicy.STATELESS` (세션 미사용) 설정을 추가.
   - 엔드포인트 권한 분리:
     - `/auth/**` ➡️ `permitAll` (인증 전 접근 가능)
     - `/admin/**` ➡️ `hasRole('ADMIN')`
     - `/staff/**` ➡️ `hasRole('STAFF')`
     - `/chat/**` ➡️ `hasRole('GUEST')`

## 🟡 2단계: 도메인 및 어댑터 설계 (AN-152, 153, 154 준비)
*인증에 필요한 핵심 도메인 모델과 DB 연동을 준비합니다.*

1. **도메인 모델**
   - 위치: `backend/src/main/java/com/anook/backend/domain/model/`
   - 내용: `Guest` (방 번호), `Staff` (PIN 번호, 이름), `Admin` (ID, PW) 엔티티 정의.
2. **Repository Port**
   - 위치: `backend/src/main/java/com/anook/backend/application/port/out/`
   - 내용: `StaffRepositoryPort` (PIN으로 직원 조회), `AdminRepositoryPort` (ID로 관리자 조회) 인터페이스 설계.
3. **Persistence Adapter**
   - 위치: `backend/src/main/java/com/anook/backend/adapter/out/persistence/`
   - 내용: 위 Port를 구현하여 실제 JPA Repository와 도메인 모델을 매핑(Mapping)하는 어댑터 작성.

## 🟠 3단계: 역할별 인증 비즈니스 로직 및 API 구현 (AN-152, 153, 154, 155)
*로그인을 검증하고 쿠키를 구워주는 실질적인 엔드포인트를 `security` 모듈에 통합하여 만듭니다.*

1. **투숙객 로직 (AN-152)**
   - Service: `GuestAuthService` (정적 QR 방번호 검증 → guest 존재 확인 → JWT 생성)
   - Controller: `GuestAuthController` ➡️ `@PostMapping("/api/auth/guest")`
   - 위치: `backend/src/main/java/com/anook/backend/security`
2. **직원 로직 (AN-153)**
   - Service: `StaffAuthService` (DB의 PIN과 일치 여부 검사 → JWT 생성)
   - Controller: `StaffAuthController` ➡️ `@PostMapping("/api/auth/staff")`
   - 위치: `backend/src/main/java/com/anook/backend/security`
3. **관리자 로직 (AN-154)**
   - Service: `AdminAuthService` (ID 확인 및 PW 해시 매칭 → JWT 생성)
   - Controller: `AdminAuthController` ➡️ `@PostMapping("/api/auth/admin")`
   - 위치: `backend/src/main/java/com/anook/backend/security`
4. **API 반환 공통 규칙 (AN-155)**
   - Controller에서 발급된 JWT를 본문(Body)이 아닌, `ResponseCookie` 객체를 사용하여 **HttpOnly, Secure 속성이 적용된 쿠키** 헤더로 반환.

## 🔴 4단계: 동시 접속 제어 로직 (AN-156)
*보안 강화를 위한 다중 기기 세션 차단 기능입니다.*

1. **저장소 활용 (Redis 또는 DB)**
   - 직원이 PIN으로 로그인하면, 발급된 토큰 고유값(JTI)을 Redis 등에 캐싱.
2. **중복 접속 차단 (`StaffAuthService`)**
   - 동일한 PIN으로 새로운 로그인이 감지되면 기존에 저장된 토큰을 무효화(블랙리스트 처리)하여, 이전 기기에서 활동하려 할 때 필터에서 차단되도록 설계.

## 🔵 5단계: 프론트엔드 연동 및 화면 구현 (AN-157)
*스프링부트에서 완성된 인증 체계를 Next.js 브라우저 화면과 연결합니다.*

1. **BFF 라우트 설정**
   - 위치: `frontend/src/app/api/auth/[...path]/route.ts`
   - 내용: 클라이언트의 로그인 요청을 Spring Boot로 프록시(Proxy). Spring Boot가 넘겨준 HttpOnly 쿠키를 브라우저에 그대로 심어주는 로직 구현.
2. **로그인 UI 컴포넌트 개발**
   - 직원 PIN 화면: `frontend/src/app/(auth)/staff/login/_components/StaffLoginForm/`
   - 관리자 로그인 화면: `frontend/src/app/(auth)/admin/login/_components/AdminLoginForm/`
   - 규칙: 페이지 최상단이 아닌, `_components/` 안에 분리하여 개발하는 규칙 적용.
3. **전역 상태 관리**
   - 위치: `frontend/src/store/useAuthStore.ts`
   - 내용: 보안상 토큰은 브라우저에서 읽지 못하므로, BFF에서 전달받은 '사용자 권한 정보(Role)와 닉네임'만 Zustand 스토어에 올려 UI 분기 처리에 활용.
