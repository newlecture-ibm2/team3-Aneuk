# 아늑(Aneuk) AI 기술 스펙 및 개발 R&R 6분할

이 문서는 아늑(Aneuk) 프로젝트의 AI 기능을 '개발 라이프사이클'과 '데이터의 흐름'에 따라 6개의 독립적인 모듈로 분할한 R&R 명세서입니다. 
팀원 6명이 각자 하나의 파이프라인을 전담하여 병렬 개발을 진행하기 가장 이상적인 구조입니다.

---

## 1. [검색] RAG Inference (고객 응대 파트) 담당자-정혜연
**"모르는 건 규정집(DB)에서 찾아서 대답해!"**
*   **역할**: 고객의 질문을 벡터화하여 가장 적절한 답변을 찾아 즉시 응대합니다. (지식 검색 및 답변)
*   **핵심 개발 요소**:
    *   `pgvector` 기반 임베딩 및 유사도 검색 API 개발
    *   유사도 0.7 미만 시 "정보 없음"으로 Fallback 하는 방어 로직
    *   실시간 고객 응대를 위한 답변 생성 속도(Latency) 최적화

## 2. [지식] RAG Ingestion (데이터 플라이휠 파트) 담당자-황서범
**"답변 못 한 질문들을 모아서 새로운 지식으로 만들어!"**
*   **역할**: AI가 답변하지 못한 질문들을 분석해 시스템의 지식을 스스로 넓혀갑니다.
*   **핵심 개발 요소**:
    *   심야 스케줄러(Cron)를 통한 미답변 질문 군집화(Clustering) 로직
    *   관리자 대시보드 내 신규 지식 승인 대기열 UI 개발
    *   관리자 승인 시 텍스트를 즉시 임베딩하여 벡터 DB에 Insert 하는 로직

## 3. [분석] NLU Task 자동 분류 (해피 패스 파트)  담당자-조영일
**"고객의 자유로운 말을 우리 시스템의 코드로 완벽하게 변환해!"**
*   **역할**: 고객의 다국어 자연어를 우리 시스템이 파싱할 수 있는 완벽한 정형 데이터로 번역 및 변환합니다.
*   **핵심 개발 요소**:
    *   다국어 One-pass 프롬프트 최적화 (언어 감지+번역+의도 파악 병합)
    *   자연어를 `HK|NORM|TOWEL|2` 형태로 강제 변환하는 토큰 및 포맷 최적화

## 4. [방어] NLU 예외 제어 (Safety & Fallback 파트)  담당자-박단희
**"개인정보는 가리고, 모르는 물건은 사람에게 넘겨서 시스템을 보호해!"**
*   **역할**: AI가 헛소리를 하거나 민감 정보를 받았을 때 시스템이 터지지 않게 안전망을 구축합니다.
*   **핵심 개발 요소**:
    *   PII(개인정보) 정규식 마스킹 (프롬프트 전송 전 선제적 방어)
    *   미등록 품목(`ETC`) 발생 시 백엔드에서 Confidence Score를 강등하여 관리자에게 에스컬레이션(수동 배정) 시키는 로직

## 5. [학습] NLU 자가 튜닝 (Self-Tuning 파트) 담당자-장혜은
**"사람이 고쳐준 오답 노트를 외워서 다음번엔 틀리지 마!"**
*   **역할**: 4번(방어)에서 튕겨나간 에러를 관리자가 고쳐주면, 그 패턴을 학습시켜 AI를 점진적으로 고도화합니다.
*   **핵심 개발 요소**:
    *   관리자의 '수동 정정 데이터(오답 노트)' DB 적재 로직
    *   다음 API 호출 시 이 데이터를 **퓨샷(Few-shot)** 예시로 프롬프트에 동적 주입하는 로직

## 6. [생성] NLG 데이터 요약 (인수인계 브리핑 파트) 담당자-강병수
**"딱딱한 시스템 데이터를 사람이 읽기 편한 따뜻한 말로 요약해 줘!"**
*   **역할**: 백엔드 DB에 쌓인 미처리 태스크 및 주요 내역들을 모아 사람(직원)이 읽기 편하게 가공합니다.
*   **핵심 개발 요소**:
    *   잔여 업무, 관리자 개입 특이사항, 주요(URGENT/HIGH) 처리 내역 데이터 취합 쿼리
    *   전문적인 호텔리어 톤앤매너를 강제하는 3~5줄 요약 프롬프트 튜닝

---

### 💡 (참고) RAG와 Few-Shot(퓨샷)의 차이점
AI를 똑똑하게 만드는 두 기술의 '목적'은 완전히 다릅니다. 이 구조를 이해하면 개발 방향성이 명확해집니다.

*   **RAG (검색 증강 생성) = 오픈북 시험**
    *   목적: AI에게 **"지식과 정보(Fact)"**를 제공할 때 사용합니다. (1번, 2번 파트)
    *   예: "조식 시간은 7시야"라는 규정을 찾아보고 고객에게 대답.
*   **Few-Shot (퓨샷) = 기출문제 예시**
    *   목적: AI에게 **"행동 패턴과 출력 규격(Rule)"**을 가르칠 때 사용합니다. (5번 파트)
    *   예: "[예시] 고객: 종이컵 줘 → 출력: FB|ETC|CUP|1" 처럼 예시를 보여주어 올바른 부서 코드를 출력하도록 유도.
