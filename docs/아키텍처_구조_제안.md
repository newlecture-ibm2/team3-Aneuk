# 아늑(Anook) 프로젝트 아키텍처 구조

> **v2.0** — 도메인 기반 병렬 AI 구조 + 공통 JSON 스키마 + 정적 QR 인증 반영
> 헥사고날 패턴 기반 설계. ERD, API 스펙은 별도 문서에서 정의합니다.

---

## 1. 시스템 배포 구조 (4-Container)

```
[Guest 모바일 / Staff 태블릿 / Admin PC]
              │ HTTPS
              ▼
┌──────────────────────────┐
│  Nginx (Reverse Proxy)   │  ← 유일한 외부 노출 포트 (80, 443)
│  - SSL 종단               │
│  - PWA 정적 파일 서빙      │
│  - /api/** → app:8080     │
│  - /ws/** → app:8080      │
└────────────┬─────────────┘
             │ 내부망
             ▼
┌──────────────────────────┐
│  Spring Boot (app)       │  ← 포트 외부 비노출
│  - REST API              │
│  - WebSocket (STOMP)     │
│  - JWT (HttpOnly Cookie)  │
└─────┬─────────────┬──────┘
      │ 내부망         │ 내부망 (HTTP)
      ▼                ▼
┌────────────┐  ┌──────────────────────────┐
│ PostgreSQL  │  │ Python AI (FastAPI)        │  ← 포트 외부 비노출
│ + pgvector  │  │                            │
│ - 관계형 DB  │  │ [Router] Gemini 클라우드     │  ← 도메인 라우팅
│ - 벡터 DB   │  │   ├── HK Agent (하우스키핑)   │
└────────────┘  │   ├── FB Agent (식음료)       │
                  │   ├── FACILITY Agent (시설)   │
                  │   ├── CONCIERGE Agent (컨시어지)│
                  │   ├── EMERGENCY Agent (긴급)  │
                  │   └── FRONT Agent (Fallback)  │
                  │ - 공통 JSON 스키마 출력        │
                  │ - RAG (pgvector) 검색         │
                  │ - Few-Shot 동적 주입           │
                  └──────────────────────────┘
```

> **핵심**: Spring Boot는 AI 로직을 전혀 모릅니다. `adapter/out/ai/`에서 Python AI 서비스를 HTTP로 호출할 뿐이며,
> AI 엔진 교체(Gemini → OpenAI 등)시 Spring Boot 코드는 수정 없이 Python 측만 변경하면 됩니다.
>
> **AI 구조 변경 (v2)**: 기술 중심 릴레이(RAG→NLU→NLG 순차)에서 **도메인 기반 병렬 구조**로 전환.
> 클라우드 모델(Gemini)이 라우팅을 담당하고, 6개 부서별 Agent가 독립적으로 JSON 응답을 생성합니다.

## 2. 백엔드 모듈 구조 (헥사고날 아키텍처)

> **핵심 원칙** (Reference 5가지 불변 원칙 준수)
> 1. Domain은 순수 POJO — JPA/Spring 어노테이션 없음
> 2. Port(Out)은 도메인 모델만 반환 — JPA Entity 반환 금지
> 3. Service는 Port를 통해서만 DB 접근 — JPA Repository 직접 참조 금지
> 4. Controller DTO와 Service DTO는 분리
> 5. 모듈 간 소통은 Port(인터페이스)를 통해서만

### 2.1 전체 패키지 맵

> **구조 원칙: 역할 경계(Bounded Context) 기반 분리**
> - **Guest (투숙객)** = 일반 유저 → 루트 레벨에 도메인 직접 배치
> - **Staff (직원)** = 서비스 수행자 → `staff/` 네임스페이스로 분리
> - **Admin (관리자)** = 서비스 관리자 → `admin/` 네임스페이스로 분리

> **모듈 간 참조 규칙:**
> 각 모듈은 자체 `domain/model/`에 순수 POJO 도메인 모델을 소유합니다.
> 다른 모듈의 데이터가 필요한 경우, **절대 다른 모듈의 domain을 직접 import하지 않고**
> 자기 모듈의 `port/out/`에 조회용 Port 인터페이스를 정의하여 접근합니다.
>
> ```
> ✅ staff/task/port/out/RequestQueryPort.java  →  request 모듈의 Adapter가 구현
> ❌ staff/task/service/ 에서 request/domain/model/Request.java 직접 import
> ```

> 실제 PMS(Opera, Fidelio 등) 연동 전까지 사용하는 **가상 PMS** 역할.
> Admin이 직접 투숙객을 체크인/아웃 처리합니다. **QR 토큰 불필요 — 정적 QR(방번호 URL) 사용.**
> **체크아웃 시 GuestSession 레코드를 삭제합니다 (Hard Delete).**
> "있으면 체류 중, 없으면 빈 방" — 상태 필터 없이 단순하게 운영합니다.
> 향후 실제 PMS 연동 시 `adapter/out/pms/`만 교체하면 됩니다.

```
admin/guest/
├── domain/model/
│   └── GuestSession.java                # 투숙객 (객실, 언어)
│       └── create(roomId, lang)         ← 체크인 (이름/토큰 불필요)
├── application/
│   ├── port/
│   │   ├── in/   → CheckInGuestUseCase, CheckOutGuestUseCase, GetGuestsUseCase
│   │   └── out/  → GuestSessionRepositoryPort
│   └── service/  → CheckInGuestService, CheckOutGuestService, GetGuestsService
└── adapter/
    ├── in/web/    → AdminGuestController
    └── out/persistence/ → GuestSessionPersistenceAdapter
```

> **Guest 인증 플로우 (정적 QR + JWT + Hard Delete):**
>
> ```
> [Admin 체크인]
>   Admin이 객실+언어 등록 → GuestSession 생성 (room_id, language)
>
> [Guest 접속]
>   객실 내 고정 QR 스캔 → /chat/302 (방번호가 URL에 내장)
>     → security/GuestAuthService: room 302에 활성 guest 레코드 존재?
>     → 존재하면 → JWT 발급 (HttpOnly Cookie) → 채팅 화면으로 직행
>     → 없으면 → 거부 (체크아웃됨 또는 빈 방)
>
> [Admin 체크아웃]
>   Admin이 체크아웃 처리 → GuestSession 삭제 (Hard Delete)
>     → 이후 QR 스캔해도 guest 없음 → JWT 검증 실패 → 접속 불가
> ```

### 2.6 공통 인프라 (infrastructure/)

> dispatch는 도메인 모듈이 아닙니다. 자체 엔티티나 비즈니스 규칙 없이, 다른 모듈이 시키는 대로 전달만 하는 인프라 어댑터입니다.
> 각 모듈은 자체 `port/out/DispatchPort`를 정의하고, `WebSocketDispatchAdapter`가 이를 구현합니다.

```
infrastructure/
└── websocket/
    └── WebSocketDispatchAdapter.java    # DispatchPort 구현체 (STOMP 발송)
```

> **사용 패턴:**
> ```java
> // request/application/port/out/DispatchPort.java
> public interface DispatchPort {
>     void send(String target, String eventType, Object payload);
> }
>
> // infrastructure/websocket/WebSocketDispatchAdapter.java
> @Component
> public class WebSocketDispatchAdapter implements DispatchPort {
>     // SimpMessagingTemplate으로 STOMP 전송
> }
> ```

> **`adapter/out/` 하위 카테고리 기준:**
> 헥사고날의 Driven Adapter는 외부 시스템 종류별로 하위 폴더를 나눕니다.
> | 폴더 | 용도 | 위치 |
> |------|------|------|
> | `persistence/` | DB (JPA, pgvector) | 각 모듈 adapter/out/ |
> | `ai/` | **Python AI 서비스 HTTP 호출** | request, message, knowledge, admin/handover |
> | `websocket/` | 실시간 통신 (STOMP) | **infrastructure/** (루트 레벨) |
> | `scheduler/` | 배치 작업 (Cron) | knowledge (미답변 클러스터링) |
>
> 향후 FCM, SMS, 외부 PMS 등이 추가되면 `infrastructure/fcm/`, `infrastructure/pms/`로 자연스럽게 확장됩니다.

## 4. 모듈 생성 체크리스트 (Anook 적용)

### 백엔드 — 새 모듈 추가 시
```
□ domain/model/ 에 순수 POJO 도메인 모델 생성
  □ JPA 어노테이션 없음
  □ 비즈니스 행위 메서드 포함 (changeStatus, validate 등)

□ application/port/in/ 에 UseCase 인터페이스 정의
  □ 네이밍: {동작}{도메인}UseCase (예: SubmitRequestUseCase)

□ application/port/out/ 에 RepositoryPort 인터페이스 정의
  □ 반환 타입이 도메인 모델인지 확인 (JPA Entity 반환 금지!)

□ application/service/ 에 UseCase 구현체 생성
  □ import에 adapter.out.persistence.* 가 없는지 확인!
  □ 단일 메서드 50줄 이하

□ adapter/in/web/ 에 Controller 생성
  □ UseCase 인터페이스에만 의존
  □ DTO는 dto/request/, dto/response/ 에 별도 파일
  □ Controller에 비즈니스 로직 없음

□ adapter/out/persistence/ 에 구현체 생성
  □ JpaEntity (JPA 어노테이션은 여기에만!)
  □ JpaRepository
  □ PersistenceAdapter (Domain ↔ Entity 매핑)
```

### 프론트엔드 — 새 페이지/컴포넌트 추가 시
```
□ page.tsx 생성 시 동시에 _components/ 폴더 생성
□ page.tsx 150줄 이하 유지
□ 복잡한 로직은 커스텀 훅(use{Feature}.ts)으로 분리
□ 컴포넌트는 폴더 구조 (ComponentName/ + .module.css + index.ts)
□ 300줄 초과 시 서브 컴포넌트 분리
□ API 함수는 도메인별 파일 (lib/api/{domain}API.ts)
```

## 5. 의존성 방향 (절대 역행 금지)

### 백엔드
```
각 모듈 (request, message, knowledge, staff/task, admin/* ...)
         │
         ├── Adapter(In/Out) → Application(Port, Service) → Domain(model/)
         │        ↓                    ↓                        ↓
         │   [컨트롤러,JPA]     [유스케이스,포트]          [순수 POJO]
         │   (바깥)              (중간)                  (안쪽)
         │
         └── ★ 각 모듈은 자체 domain/model/ 소유 (공유 domain 패키지 없음)
             ★ 다른 모듈 데이터 필요 시 port/out/ 인터페이스로 조회 (직접 import 금지)
             ★ staff/task → RequestQueryPort (O)
             ★ staff/task → request/domain/model/Request (X 절대 금지!)
```

### 프론트엔드
```
page.tsx → _components/ → hooks/ → lib/api/ → BFF(api/[...path]) → Spring Boot
```
