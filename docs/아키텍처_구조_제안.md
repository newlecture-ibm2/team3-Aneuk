# 아늑(Aneuk) 프로젝트 아키텍처 구조

> `PROJECT_ARCHITECTURE_REFERENCE.md`의 헥사고날 패턴과 NCafe 교훈을 기반으로 설계한 Aneuk MVP 프로젝트 구조입니다.
> ERD, API 스펙은 별도 문서에서 정의합니다.

## 1. 시스템 배포 구조 (4-Container)

```
[Guest 모바일 / Staff 태블릿 / Admin PC]
              │ HTTPS
              ▼
┌──────────────────────────┐
│  Nginx (Reverse Proxy)   │  ← 유일한 외부 노출 포트 (80, 443)
│  - SSL 종단               │
│  - PWA 정적 파일 서빙      │
│  - /api/** → app:8080     │
│  - /ws/** → app:8080      │
└────────────┬─────────────┘
             │ 내부망
             ▼
┌──────────────────────────┐
│  Spring Boot (app)       │  ← 포트 외부 비노출
│  - REST API              │
│  - WebSocket (STOMP)     │
│  - JWT (HttpOnly Cookie)  │
└─────┬─────────────┬──────┘
      │ 내부망         │ 내부망 (HTTP)
      ▼                ▼
┌────────────┐  ┌─────────────┐
│ PostgreSQL  │  │ Python AI     │
│ + pgvector  │  │ (FastAPI)     │  ← 포트 외부 비노출
│ - 관계형 DB  │  │ - Gemini API  │
│ - 벡터 DB   │  │ - NLU 분석    │
└────────────┘  │ - RAG 검색    │
                  │ - 임베딩      │
                  │ - NLG 요약    │
                  └─────────────┘
```

> **핵심**: Spring Boot는 AI 로직을 전혀 모릅니다. `adapter/out/ai/`에서 Python AI 서비스를 HTTP로 호출할 뿐이며,
> AI 엔진 교체(Gemini → OpenAI 등)시 Spring Boot 코드는 수정 없이 Python 측만 변경하면 됩니다.

## 2. 백엔드 모듈 구조 (헥사고날 아키텍처)

> **핵심 원칙** (Reference 5가지 불변 원칙 준수)
> 1. Domain은 순수 POJO — JPA/Spring 어노테이션 없음
> 2. Port(Out)은 도메인 모델만 반환 — JPA Entity 반환 금지
> 3. Service는 Port를 통해서만 DB 접근 — JPA Repository 직접 참조 금지
> 4. Controller DTO와 Service DTO는 분리
> 5. 모듈 간 소통은 Port(인터페이스)를 통해서만

> Spring Boot의 rag 모듈은 관리자 지식 CRUD + Python 서비스 호출 중계만 담당합니다.

```
rag/

│   ├── port/in/   → SearchKnowledgeUseCase, ApproveKnowledgeUseCase

```
handover/
├── domain/model/

│   ├── port/in/   → GenerateHandoverUseCase
│   ├── port/out/  → HandoverAiPort, RequestQueryPort (request 모듈 조회용)
│   └── service/   → GenerateHandoverService
└── adapter/

#### notification 모듈
```
notification/
├── domain/model/
│   └── Notification.java                # 알림 도메인 (대상, 유형, 메시지, 읽음 여부)
├── application/
│   ├── port/in/   → SendNotificationUseCase
│   ├── port/out/  → NotificationRepositoryPort, NotificationSendPort
│   └── service/   → SendNotificationService
└── adapter/
    └── out/
        ├── persistence/ → NotificationPersistenceAdapter (알림 이력 DB 저장)
        └── websocket/   → WebSocketNotificationAdapter  ← NotificationSendPort 구현
```

> **`adapter/out/` 하위 카테고리 기준:**
> 헥사고날의 Driven Adapter는 외부 시스템 종류별로 하위 폴더를 나눕니다.
> | 폴더 | 용도 | 사용 모듈 |
> |------|------|----------|
> | `persistence/` | DB (JPA, pgvector) | 모든 모듈 |
> | `ai/` | **Python AI 서비스 HTTP 호출** (Gemini 직접 호출 아님!) | request, chat, rag, handover |
> | `websocket/` | 실시간 통신 (STOMP) | notification |
> | `scheduler/` | 배치 작업 (Cron) | rag (미답변 클러스터링) |
>
> 향후 FCM, SMS, 외부 PMS 등이 추가되면 `adapter/out/fcm/`, `adapter/out/pms/`로 자연스럽게 확장됩니다.

## 4. 모듈 생성 체크리스트 (Aneuk 적용)

### 백엔드 — 새 모듈 추가 시
```
□ domain/model/ 에 순수 POJO 도메인 모델 생성
  □ JPA 어노테이션 없음
  □ 비즈니스 행위 메서드 포함 (changeStatus, validate 등)

□ application/port/in/ 에 UseCase 인터페이스 정의
  □ 네이밍: {동작}{도메인}UseCase (예: SubmitRequestUseCase)

□ application/port/out/ 에 RepositoryPort 인터페이스 정의
  □ 반환 타입이 도메인 모델인지 확인 (JPA Entity 반환 금지!)

□ application/service/ 에 UseCase 구현체 생성
  □ import에 adapter.out.persistence.* 가 없는지 확인!
  □ 단일 메서드 50줄 이하

□ adapter/in/web/ 에 Controller 생성
  □ UseCase 인터페이스에만 의존
  □ DTO는 dto/request/, dto/response/ 에 별도 파일
  □ Controller에 비즈니스 로직 없음

□ adapter/out/persistence/ 에 구현체 생성
  □ JpaEntity (JPA 어노테이션은 여기에만!)
  □ JpaRepository
  □ PersistenceAdapter (Domain ↔ Entity 매핑)
```

### 프론트엔드 — 새 페이지/컴포넌트 추가 시
```
□ page.tsx 생성 시 동시에 _components/ 폴더 생성
□ page.tsx 150줄 이하 유지
□ 복잡한 로직은 커스텀 훅(use{Feature}.ts)으로 분리
□ 컴포넌트는 폴더 구조 (ComponentName/ + .module.css + index.ts)
□ 300줄 초과 시 서브 컴포넌트 분리
□ API 함수는 도메인별 파일 (lib/api/{domain}API.ts)
```

---

## 5. 의존성 방향 (절대 역행 금지)

### 백엔드
```
Adapter(In/Out) → Application(Port, Service) → Domain
        ↓                    ↓                    ↓
   [컨트롤러,JPA]     [유스케이스,포트]       [순수 POJO]
   (바깥)              (중간)                (안쪽)

★ 항상 바깥 → 안쪽으로만! Domain이 Adapter를 import하면 즉시 수정!
```

### 프론트엔드
```
page.tsx → _components/ → hooks/ → lib/api/ → BFF(api/[...path]) → Spring Boot
```

## 6. 네이밍 규칙 통일

### 백엔드
| 대상 | 규칙 | 예시 |
|------|------|------|
| UseCase | `{동작}{도메인}UseCase` | `SubmitRequestUseCase` |
| Service | `{동작}{도메인}Service` | `SubmitRequestService` |
| Port(Out) | `{도메인}RepositoryPort` | `RequestRepositoryPort` |
| Controller | `{역할}{도메인}Controller` | `AdminRequestController` |
| Adapter | `{도메인}PersistenceAdapter` | `RequestPersistenceAdapter` |
| Command | `{동작}{도메인}Command` | `SubmitRequestCommand` |
| Result | `{동작}{도메인}Result` | `GetRequestDetailResult` |

### 프론트엔드
| 대상 | 규칙 | 예시 |
|------|------|------|
| 페이지 | `app/{route}/page.tsx` | `app/staff/dashboard/page.tsx` |
| 컴포넌트 | `{Name}/{Name}.tsx` | `RequestCard/RequestCard.tsx` |
| 커스텀 훅 | `use{Feature}.ts` | `useRequests.ts` |
| API 함수 | `{domain}API.ts` | `requestAPI.ts` |
| 전역 상태 | `use{Domain}Store.ts` | `useUiStore.ts` |
