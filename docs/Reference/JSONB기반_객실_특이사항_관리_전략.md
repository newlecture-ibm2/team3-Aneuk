# [Host 서비스] 객실 특이사항(Remarks) 설계 전략 및 방향성

## 1. 데이터 저장 전략: JSONB 컬럼 방식
실시간 업무 자동화 레이어인 'Host' 서비스의 특성을 고려하여, 별도 테이블 분리 대신 `rooms` 테이블 내 단일 `jsonb` 컬럼 저장을 채택함.

### 선정 이유 (Architecture Decision)
* **관리 효율성**: Hard Delete 기반의 가상 PMS 환경에서 부모-자식 관계 유지 비용보다 단일 레코드 관리의 이점이 큼.
* **낮은 결합도**: 실 PMS의 데이터 구조가 변경되어도 DB 스키마 변경(Alter Table) 없이 어플리케이션 레이어에서 유연하게 대응 가능.
* **조회 성능**: Join 연산 없이 단일 Index Scan만으로 객실의 모든 상태 값을 즉시 로드 가능.

## 2. 내부 데이터 구조: Array of Objects (List)
단순 Key-Value(Object)가 아닌 리스트 구조를 채택하여 데이터 누락 방지 및 태스크 확장성 확보.

* **중복 허용**: 동일 카테고리(예: 청소)의 요청이 여러 건 발생해도 덮어쓰기 없이 모두 수용 가능.
* **태스크화 용이**: 각 배열 요소를 독립적인 업무 단위(Task)로 간주하여 자동화 로직(Loop 처리)에 투입하기 적합.
* **메타데이터 확장**: 개별 요청별로 생성 시간, 긴급도, 처리 상태 등의 필드를 자유롭게 추가 가능.

## 3. 실시간 업무 자동화(Automation) 연동 방향
* **이벤트 트리거**: `jsonb` 내부의 특정 `type`이나 `isUrgent` 필드를 감지하여 담당 부서(미화, 시설 등)에 실시간 알림 발송.
* **필터링 최적화**: PostgreSQL의 GIN 인덱스를 활용하여, 데이터가 JSON 내부에 있더라도 "긴급 건이 포함된 객실"만 DB 레벨에서 고속 필터링.

## 4. 프론트엔드 및 API 연동 방향
* **Zero Parsing**: 백엔드에서 가공된 JSON을 그대로 전달함으로써 Next.js 프론트엔드에서 별도의 데이터 파싱 로직 없이 즉시 UI 렌더링.
* **상태 동기화**: 체크아웃 시 해당 객실 레코드를 Hard Delete 함으로써 가상 PMS와 Host 서비스 간의 데이터 정합성을 단순하고 명확하게 유지.
