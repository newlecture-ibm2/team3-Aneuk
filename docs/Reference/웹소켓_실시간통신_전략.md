# 실시간 통신 전략 (WebSocket / STOMP)

> Aneuk 프로젝트의 서버 → 클라이언트 실시간 Push 구조 및 채널 설계

---

## 1. 왜 WebSocket인가

Aneuk은 **다수의 역할(Guest/Staff/Admin)에게 서로 다른 실시간 이벤트를 동시에 Push**해야 합니다.

| 대안 | 장점 | Aneuk에 부적합한 이유 |
|------|------|---------------------|
| **Polling** | 구현 가장 쉬움 | 1초마다 반복 조회 → 불필요한 트래픽, 체감 지연 |
| **SSE** | 단방향 Push 간단 | 채널 분리(부서별/객실별) 어려움, 양방향 불가 |
| **WebSocket** | 양방향, 채널 분리 자유 | ✅ 채택 — 한 번 구축하면 모든 실시간 이벤트에 재활용 |

> ChatGPT/Gemini 웹 UI는 SSE를 사용하지만, 그들은 "AI 응답 스트리밍" 단일 용도입니다.
> Aneuk은 채팅 응답 + 태스크 배정 + 상태 변경 + 에스컬레이션 등 **다채널 실시간 이벤트**가 핵심이므로 WebSocket이 정석입니다.

## 2. 통신 구조 전체도

```
[Guest 브라우저]  [Staff 태블릿]  [Admin PC]
      │                │              │
      │ wss://         │ wss://       │ wss://
      ▼                ▼              ▼
┌──────────────────────────────────────────┐
│              Nginx (:443)                │
│  - /ws/** → app:8080 (WebSocket 직행)    │
│  - 그 외 → next:3000 (BFF)              │
└──────────────────┬───────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────┐
│         Spring Boot (:8080)              │
│                                          │
│  WebSocketConfig (STOMP Broker)          │
│  ├── /ws          ← 클라이언트 연결 엔드포인트  │
│  ├── /topic/**    ← 구독 채널 (서버→클라이언트) │
│  └── /app/**      ← 발행 채널 (클라이언트→서버) │
│                                          │
│  WebSocketDispatchAdapter                │
│  └── SimpMessagingTemplate.convertAndSend│
└──────────────────────────────────────────┘
```

### 핵심 포인트
- **WebSocket은 Nginx를 거쳐 Spring Boot로 직행**합니다 (BFF를 거치지 않음)
- REST API는 `Nginx → Next.js(BFF) → Spring Boot` 3단계지만, WebSocket은 `Nginx → Spring Boot` 2단계
- 이유: WebSocket은 장시간 연결 유지 프로토콜이라 BFF에서 프록시하면 복잡도만 증가

## 3. STOMP 채널 설계

### 3.1 구독 채널 (서버 → 클라이언트)

| 채널 | 구독 대상 | 이벤트 | 설명 |
|------|----------|--------|------|
| `/topic/room/{roomNo}` | Guest | `AI_RESPONSE` | AI 분석 완료 → 응답 메시지 Push |
| `/topic/room/{roomNo}` | Guest | `STATUS_CHANGED` | 내 요청 상태 변경 (수락됨, 완료됨) |
| `/topic/room/{roomNo}` | Guest | `STAFF_REPLY` | 직원이 보낸 응답 메시지 |
| `/topic/dept/{deptCode}` | Staff | `NEW_REQUEST` | 새 태스크 배정 알림 |
| `/topic/dept/{deptCode}` | Staff | `TASK_UPDATED` | 동료가 태스크 수락/완료 (목록 갱신용) |
| `/topic/admin` | Admin | `ESCALATED` | 에스컬레이션 발생 알림 |
| `/topic/admin` | Admin | `SYSTEM_ALERT` | 긴급 상황 감지 알림 |

### 3.2 채널 네이밍 규칙

```
/topic/{범위}/{식별자}
```

- `room/{roomNo}` — 객실 단위 격리 (Guest는 자기 방만 구독)
- `dept/{deptCode}` — 부서 단위 격리 (Staff는 자기 부서만 구독)
- `admin` — 관리자 전용 (단일 채널)

### 3.3 실시간 이벤트 흐름도

> **누가 행동하면 → 누구에게 Push되는가** 전체 시나리오

#### 시나리오 1: 고객 메시지 → AI 응답 → 태스크 생성

```
① Guest가 "수건 2장 주세요" 전송
   → [REST] POST /chat/302/messages → 202 즉시 응답

② Spring Boot가 AI 호출 (내부 HTTP)
   → AI 분석 완료 (domainCode: HK)

③ 결과 Push (동시 다발)
   → /topic/room/302     [AI_RESPONSE]   → Guest: "수건 2장 보내드리겠습니다" 표시
   → /topic/dept/HK      [NEW_REQUEST]   → HK 부서 Staff: 새 태스크 카드 등장
   → /topic/admin         [ESCALATED]     → Admin: confidence < 0.7이면 에스컬레이션 알림
```

#### 시나리오 2: 직원 수락 → 투숙객 상태 업데이트

```
① Staff가 태스크 수락
   → [REST] PATCH /staff/tasks/100/accept → 200 응답

② Push (동시 다발)
   → /topic/room/302     [STATUS_CHANGED]  → Guest: 상태 카드 "처리 중" 으로 변경
   → /topic/dept/HK      [TASK_UPDATED]    → 같은 부서 Staff: 목록에서 해당 태스크 갱신
```

#### 시나리오 3: 직원 완료 → 투숙객 완료 알림

```
① Staff가 태스크 완료 처리
   → [REST] PATCH /staff/tasks/100/complete → 200 응답

② Push (동시 다발)
   → /topic/room/302     [STATUS_CHANGED]  → Guest: 상태 카드 "완료됨" 으로 변경
   → /topic/dept/HK      [TASK_UPDATED]    → 같은 부서 Staff: 목록에서 완료 처리 반영
```

#### 시나리오 4: 직원이 투숙객에게 메시지 응답

```
① Staff가 한국어로 응답 작성
   → [REST] POST /staff/tasks/100/respond → 200 (번역 포함)

② Push
   → /topic/room/302     [STAFF_REPLY]     → Guest: 번역된 메시지가 채팅에 추가
```

#### 시나리오 5: 관리자 재배정

```
① Admin이 부서 재배정 (HK → FACILITY)
   → [REST] PATCH /admin/tasks/100/reassign → 200 응답

② Push (동시 다발)
   → /topic/dept/HK        [TASK_UPDATED]   → 기존 부서 Staff: 목록에서 태스크 제거
   → /topic/dept/FACILITY   [NEW_REQUEST]    → 새 부서 Staff: 새 태스크 카드 등장
   → /topic/room/302        [STATUS_CHANGED] → Guest: "다른 부서로 배정됨" 상태 표시
```

## 4. 이벤트 Payload 규격

모든 WebSocket 메시지는 동일한 래핑 구조를 사용합니다.

```json
{
  "type": "AI_RESPONSE",
  "timestamp": "2026-05-10T14:30:03",
  "payload": { ... }
}
```

### 4.1 AI 응답 도착

```json
{
  "type": "AI_RESPONSE",
  "timestamp": "2026-05-10T14:30:03",
  "payload": {
    "messageId": 42,
    "senderType": "AI",
    "content": "네, 수건 2장을 객실로 보내드리겠습니다.",
    "translatedContent": "Sure, we'll send 2 towels to your room.",
    "requestId": 100
  }
}
```

### 4.2 요청 상태 변경

```json
{
  "type": "STATUS_CHANGED",
  "timestamp": "2026-05-10T14:32:00",
  "payload": {
    "requestId": 100,
    "previousStatus": "PENDING",
    "newStatus": "IN_PROGRESS",
    "assignedStaffName": "김철수"
  }
}
```

### 4.3 새 태스크 배정 (Staff)

```json
{
  "type": "NEW_REQUEST",
  "timestamp": "2026-05-10T14:30:05",
  "payload": {
    "taskId": 100,
    "summary": "수건 2장 요청",
    "roomNumber": "302",
    "priority": "NORMAL",
    "domainCode": "HK"
  }
}
```

### 4.4 에스컬레이션 알림 (Admin)

```json
{
  "type": "ESCALATED",
  "timestamp": "2026-05-10T14:30:10",
  "payload": {
    "requestId": 100,
    "reason": "미등록 품목(ETC) 감지",
    "confidence": 0.55,
    "roomNumber": "302",
    "rawText": "종이컵 3개 주세요"
  }
}
```

## 5. 백엔드 구현 패턴 (헥사고날)

### 5.1 각 모듈은 자체 DispatchPort를 정의

```java
// request/application/port/out/DispatchPort.java
public interface DispatchPort {
    void send(String channel, String eventType, Object payload);
}
```

### 5.2 공통 인프라 어댑터가 구현

```java
// infrastructure/websocket/WebSocketDispatchAdapter.java
@Component
public class WebSocketDispatchAdapter implements DispatchPort {

    private final SimpMessagingTemplate messagingTemplate;

    @Override
    public void send(String channel, String eventType, Object payload) {
        Map<String, Object> message = Map.of(
            "type", eventType,
            "timestamp", Instant.now().toString(),
            "payload", payload
        );
        messagingTemplate.convertAndSend(channel, message);
    }
}
```

### 5.3 Service에서 사용

```java
// request/application/service/SubmitRequestService.java
@Service
public class SubmitRequestService implements SubmitRequestUseCase {

    private final DispatchPort dispatchPort;

    public void submit(SubmitRequestCommand command) {
        // ... 요청 저장, AI 호출 등 ...

        // Guest에게 AI 응답 Push
        dispatchPort.send(
            "/topic/room/" + command.getRoomNo(),
            "AI_RESPONSE",
            aiResponsePayload
        );

        // 담당 부서 Staff에게 새 태스크 Push
        dispatchPort.send(
            "/topic/dept/" + domainCode,
            "NEW_REQUEST",
            newTaskPayload
        );
    }
}
```

## 6. 프론트엔드 연결 패턴

### 6.1 WebSocket 연결 훅

```ts
// app/chat/[roomNo]/useWebSocket.ts
'use client';
import { useEffect, useRef, useCallback } from 'react';
import { Client } from '@stomp/stompjs';

interface UseWebSocketOptions {
  channel: string;
  onMessage: (data: any) => void;
}

export function useWebSocket({ channel, onMessage }: UseWebSocketOptions) {
  const clientRef = useRef<Client | null>(null);

  useEffect(() => {
    const client = new Client({
      brokerURL: `wss://${window.location.host}/ws`,
      reconnectDelay: 2000,        // 끊기면 2초 후 재연결
      heartbeatIncoming: 10000,
      heartbeatOutgoing: 10000,
    });

    client.onConnect = () => {
      client.subscribe(channel, (message) => {
        const data = JSON.parse(message.body);
        onMessage(data);
      });
    };

    client.activate();
    clientRef.current = client;

    return () => { client.deactivate(); };
  }, [channel, onMessage]);
}
```

### 6.2 페이지에서 사용

```tsx
// app/chat/[roomNo]/chat/page.tsx
function ChatPage({ params }: { params: { roomNo: string } }) {
  const { messages, addMessage } = useChat();

  const handleWsMessage = useCallback((data: any) => {
    if (data.type === 'AI_RESPONSE') {
      addMessage(data.payload);   // 메시지 목록에 추가
    }
    if (data.type === 'STATUS_CHANGED') {
      // 요청 상태 갱신
    }
  }, [addMessage]);

  useWebSocket({
    channel: `/topic/room/${params.roomNo}`,
    onMessage: handleWsMessage,
  });

  return <ChatMessages messages={messages} />;
}
```

## 7. 재연결 및 누락 방지 전략

### 끊김 감지 → 자동 재연결
- STOMP 클라이언트의 `reconnectDelay`로 지수 백오프 재연결 (2s → 4s → 8s)
- 하트비트(10초 간격)로 연결 상태 모니터링

### 누락 메시지 복구
- 재연결 성공 시, **마지막 수신 메시지 ID 기준으로 REST API를 통해 누락분 조회**
- WebSocket은 실시간 Push 전용, 데이터 정합성은 REST API가 보장

```
[재연결 성공]
  → GET /api/chat/302/messages?after={lastMessageId}
  → 누락된 메시지들을 목록에 병합
```

## 8. Nginx WebSocket 프록시 설정

```nginx
# /ws/** → Spring Boot WebSocket 직행
location /ws/ {
    proxy_pass http://app:8080;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
    proxy_read_timeout 86400s;   # 24시간 (장시간 연결 유지)
}
```

> `proxy_read_timeout`을 충분히 길게 설정하지 않으면 Nginx가 일정 시간 후 WebSocket 연결을 강제로 끊습니다.
