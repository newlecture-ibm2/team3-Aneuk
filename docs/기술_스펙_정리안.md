# 아늑(Aneuk) 기술 스펙 및 아키텍처 명세서

> 프론트엔드부터 인프라까지, Aneuk MVP 개발을 위한 기술 표준, 아키텍처, 그리고 주요 기술적 난제 해결 전략을 정의한 문서입니다.

---

## 1. 시스템 아키텍처 개요

Aneuk은 도메인의 복잡성(AI 분석, 다수의 부서 라우팅, 실시간 통신)을 효과적으로 관리하기 위해 **헥사고날 아키텍처(Hexagonal Architecture)** 기반의 설계 사상을 채택합니다.

*   **외부 의존성 격리**: AI 엔진(Gemini), 알림 시스템(WebSocket/FCM), 데이터베이스(PostgreSQL)를 Port/Adapter로 완전히 분리하여, 향후 다른 AI 스펙으로 교체하거나 알림 방식을 변경할 때 도메인 로직에 영향이 없도록 설계.
*   **분산 환경 고려**: 추후 PMS 및 IoT 확장에 대비하여 핵심 요청 루프와 부가 서비스를 유연하게 분리할 수 있는 기반 확보.

### Challenge 2. 오프라인 퍼스트 (Offline-First) UX 보장 및 액션 이원화
*   **문제 상황**: 직원이 엘리베이터나 수건 보관실 등 데드존(인터넷 단절 구간)에서 앱 조작 시 프리징이 걸리면 시스템 신뢰성이 하락. 반면 무조건 오프라인 액션을 허용하면 다수가 동시에 업무를 수락하는 '헛걸음' 오류 발생.
*   **해결 전략 (액션 성격에 따른 이원화)**:
    *   **독점 액션(완료) - Optimistic UI 적용**: 이미 내게 배정되어 경쟁자가 없는 업무의 '완료' 처리는 오프라인에서도 클릭 즉시 시각적으로 완료 처리. 로컬 IndexedDB에 작업 내역을 적재 후 온라인 복귀 시 Background Sync로 서버와 동기화.
    *   **경쟁 액션(수락) - 오프라인 차단**: 여러 직원이 다툴 수 있는 미배정 태스크의 '수락' 조작은 오프라인 진입 시 즉각 비활성화시키고 "네트워크 연결 불안정" 팝업 노출. 철저하게 서버 통신을 강제하여 헛걸음 방지.

### Challenge 3. AI 분석 체감 지연(Latency) 극복
*   **문제 상황**: Gemini API 외부 호출 대기 시간이 2~4초 가량 소요. 이 데드 타임 동안 고객이 버튼을 연타하거나 버그로 인식.
*   **해결 전략**:
    *   **Asynchronous Event (비동기 처리)**: 컨트롤러 단에서 고객의 메시지를 받으면 즉시 HTTP 200 OK("요청 접수완료") 리턴 + 디바운스를 통한 연타 차단. 
    *   **Skeleton/Typing UI**: 프론트 화면에서는 즉시 "AI가 확인하고 있습니다..." 타이핑 애니메이션 실행.
    *   **Event Publisher**: 백엔드는 스레드에서 AI 결과를 받은 후 WebSocket 채널에 정답(태스크 정보)을 Broadcast.

### Challenge 4. 분산 환경의 다중 접속 상태 경합 (Concurrency)
*   **문제 상황**: 여러 명의 직원이 복도(온라인)에서 1개의 "수건 요청"을 보고 동시에 "수락(진행중)" 버튼을 누르는 경우 발생하는 중복 배정 문제.
*   **해결 전략**:
    *   **도메인 레벨 낙관적 락(Optimistic Locking) 적용**: `Request` 테이블에 `@Version` 필드 삽입.
    *   동시에 DB 트랜잭션이 들어올 경우 먼저 진입한 트랜잭션만 성공(Version+1), 두 번째 트랜잭션은 `OptimisticLockException` 발생 처리. 
    *   늦게 클릭한 직원의 클라이언트 앱에는 "이미 다른 직원이 배정받았습니다"라는 에러 응답을 내리고 즉시 태스크 리스트를 최신화(갱신)함.

### Challenge 5. 혼합형 다국어 (i18n vs AI API)
*   **문제 상황**: 모든 시스템 메시지를 AI 번역기에 태우면 불필요한 API 호출 비용과 딜레이 발생.
*   **해결 전략**:
    *   **동적 리소스 (고객 생 발화, 직원 메모)**: 무조건 Gemini AI를 통해 번역 및 분석 처리. DB에 `content_org`와 `content_trans`로 이중 저장.
    *   **정적 시스템 리소스 (버튼 이름, 상태값 텍스트)**: 프론트엔드 영역의 다국어 라이브러리(`react-i18next` 등)로 분리 통제. 예를 들어, 상태값 `COMPLETED`는 사용자의 세션 언어팩(ko/en)에 따라 로컬 브라우저 단에서 매핑('완료/Done').

### Challenge 6. 업무자 인증 및 계정 관리 보안 (Security & Auth)
*   **문제 상황**: 직원용 회원가입 페이지를 열어둘 경우 외부인 접근 리스크가 있으며, 퇴사자 발생 시 관리에 공백이 생길 우려. 또한, 한 명의 코드로 여러 기기에서 동시 접속하면 책임 소재(Audit Trail) 추적이 무의미해짐.
*   **해결 전략**:
    *   **관리자 중앙 통제 (Admin-Provisioning)**: 직원용 가입 페이지는 존재하지 않음. 오직 관리자(Admin)만이 '업무자 관리 페이지'에서 신규 직원을 등록하고 **개인별 고유 코드(PIN)**를 발급.
    *   **퇴사자 접근 즉시 차단**: 관리자가 전체 직원 리스트에서 퇴사자의 상태를 '비활성화(Blocked)'로 변경하는 즉시 해당 코드의 로그인 권한 박탈.
    *   **다중 기기 로그인 제한 (Concurrent Session Control)**: 백엔드에서 동일 코드로 새로운 기기(JWT 갱신)가 접속하면 기존 기기의 접속 상태를 강제 만료 처리하여 **1코드 1로그인 기기 원칙**을 강제함. 코드를 타인과 공유하는 행위를 시스템 레벨에서 원천 차단.

### Challenge 7. 프론트엔드 상태 관리 최적화 (Server State 동기화 및 로컬 상태 경량화)
*   **문제 상황**: 실시간 WebSocket으로 다량의 업무 할당 및 상태 변경 이벤트가 쏟아질 때, 프론트엔드 내부에 서버 데이터와 UI 로컬 상태(모달 등)가 얽히면 불필요한 렌더링이 발생하고 오프라인 복구 시 상태 충돌 및 복잡도가 급증함.
*   **해결 전략 (상태 역할의 엄격한 분리)**:
    *   **Server State (React Query)**: 백엔드 DB와 연동되는 '진짜' 데이터(업무 리스트, 알림 히스토리 등)의 캐싱과 비동기 동기화를 전담. WebSocket을 통해 특정 업무의 상태 변경(예: '수락' 이벤트) 메시지를 수신하면, 전체 리스트를 재호출하는 대신 React Query의 특정 캐시만 부분 업데이트(Optimistic Update)하거나 무효화(Invalidate)하여 네트워크 트래픽을 최소화하고 화면을 즉시 갱신.
    *   **Local State (Zustand)**: UI 컴포넌트의 순수 클라이언트 상태(현재 선택된 탭, 모달 표시 여부, 폼 입력 데이터 등)만 가볍게(Lightweight) 전역 관리. 무거운 상태 관리자(Redux 등)를 배제하고 서버 데이터를 React Query로 이관함으로써, 클라이언트의 상태 트리를 극도로 가볍게 유지하여 모바일 디바이스(PWA)에서의 메모리 점유율을 최적화.
    *   **오프라인 동기화 큐 결합**: 네트워크 단절 시 발생한 '완료' 처리 액션 등은 Zustand 상태에 임시 반영하고 세부 데이터는 IndexedDB에 적재. 온라인 복귀 시 React Query의 Mutation 큐를 통해 백그라운드에서 순차적으로 서버 API를 호출하여 WebSocket 히스토리와 최종 동기화(Sync) 수행.

### Challenge 8. 데이터 프라이버시 및 개인정보(PII) 보호 (성능 최적화 포함)
*   **문제 상황**: 고객이 요청사항(채팅)에 휴대전화 번호, 신용카드 번호 등 민감한 개인정보(PII)를 포함할 경우, 이를 외부 AI(Gemini)로 그대로 전송하면 정보 유출 우려가 있음. 반면, 안전을 위해 무거운 암호화 API(DLP 등)를 거치면 AI 응답 지연 시간(Latency)이 크게 증가하여 서비스 체감 품질이 하락함.
*   **해결 전략 (지연 시간 Zero의 하이브리드 방어)**:
    *   **1차 방어 (초경량 정규표현식 마스킹)**: 백엔드(Spring Boot) 단에서 패턴이 명확한 핵심 개인정보(전화번호, 계좌번호, 주민/여권번호 등)를 정규표현식(Regex)으로 즉시 찾아내 `[개인정보]`로 치환. 처리 속도가 1밀리초(0.001초) 미만으로 지연 시간이 전혀 발생하지 않음.
    *   **2차 방어 (Enterprise AI 약관 활용)**: B2B 기업용 API(Google Cloud Vertex AI 등)를 활용하여 "입력 데이터를 모델 학습(Training)에 절대 사용하지 않는다"는 시스템적/법적 격리를 보장받음. 정규식 필터를 빠져나간 비정형 이름 등이 API로 전송되더라도 안전하게 보호됨.
    *   **UX 설계 방어 (입력 폼 분리)**: 결제나 본인 인증 등 실제 PII가 필요한 상황에서는 자유 텍스트 창을 막고 전용 입력 폼(보안 모달)을 띄우는 UX로 유도하여 원천적인 입력 리스크를 제거함.

## 4. AI 토큰 최적화 및 태스크 코드화(Task Code) 아키텍처

AI API 비용(토큰)을 극단적으로 줄이면서도 백엔드 시스템의 안정성을 확 끌어올릴 수 있는 프로젝트 핵심 아키텍처입니다. Function Calling의 무거움을 우회하고 자연어를 "코드화"하여 DB와 통신합니다.

### 4.1 One-pass 프로세싱 (API 단일 호출)
*   **개념**: 다국어 처리 시 번역 API와 분석 API를 두 번 호출하는 구조를 탈피합니다.
*   **전략**: 
    *   언어 감지 라이브러리(`langdetect`)조차 사용하지 않고 로직을 단순화합니다. (짧은 고객 발화문 특성 상 라이브러리로 사전 분류하는 것은 과도한 최적화(Over-engineering)이며, 에러 포인트만 늘릴 우려가 있음).
    *   하나의 Prompt로 묶어 처리: **"입력 언어와 무관하게 핵심 의도(Intent)와 식별자(Entity)만 추출해서 무조건 한국어 기준으로 변환해"**라고 지시합니다.

### 4.2 출력 토큰 초경량화 (자연어 → 코드 변환)
*   **개념**: 무거운 구조화 도구(Function Calling JSON) 대신, 특수 구분자(`|`)를 이용한 포맷팅을 강제하여 AI가 **약속된 코드 모델**만 응답하도록 통제합니다.
*   **규격 예시**: `[부서별칭]|[긴급도]|[분류코드]|[상세항목]|[수량]`
    *   *AS-IS (약 40토큰)*: "담당 부서: 객실 관리팀, 긴급도: 보통, 분류: 비품요청, 품목: 수건 2장"
    *   *TO-BE (약 8토큰)*: **`HK|NORM|REQ_ITEM|TOWEL|2`**
*   **RAG 참조 최적화**: 위반 규정 등 RAG 문서 검토 시, 긴 문장 대신 "정책 ID(`P-01`)"만 출력하도록 지시하고 클라이언트 표시 문구는 서버가 내부 DB로 조립합니다.
*   **기대 효과**: API 출력(Output) 토큰 사용량 **80~90% 획기적 절감** 및 백엔드의 자연어 파싱 에러 제로화.

### 4.3 유연한 확장성 (DB Definitions + Enum 분기)
> **※ 성격 (AI 오류 방어)**: AI가 규격에 없는 이상한 코드를 뱉어내는 **'시스템 오류(할루시네이션)'**를 백엔드에서 문지기처럼 차단하는 기술적 방어막입니다.
*   **DB 설계 (`Task_Definitions`)**:
    *   `REQ_ITEM`(비품), `FIX_ENGR`(보수), `EMER_SITU`(긴급) 등의 코드를 유연성 높은 테이블 구조로 관리합니다. 향후 발렛, 스파 예약 등 신규 업무가 생겨도 소스 수정 없이 데이터(레코드) 추가만으로 자동 라우팅 확장이 가능합니다.
*   **백엔드 (안정성 필터)**:
    *   AI가 넘겨준 문자열 배열(`split('|')`) 값을 백엔드에서 **Enum**으로 캐스팅(`TaskType.valueOf()`)하여 강타입(Type Safety)을 확보합니다.
    *   AI 할루시네이션(미정의 코드 생성) 발생 시, 즉각 `UNDEFINED` 타입으로 치환하여 관리자의 수동 처리(Fallback) 큐로 안전하게 이관합니다.

### 4.4 미등록 품목(Edge Case) 처리와 하이브리드 워크플로우 연동
> **※ 성격 (비즈니스 예외 처리)**: AI는 정상 작동했으나, 고객이 호텔 코드에 없는 특이한 물건을 요구했을 때 관리자에게 검토를 올리도록 기획된 **'정상적인 예외 처리 과정'**입니다.
*   **문제 상황**: "종이컵"이나 "가습기" 등 코드 테이블에 사전 정의되지 않은 비표준 특이 요청.
*   **AI의 `ETC` 라우팅**: 프롬프트에 "정의되지 않은 품목은 `ETC` 코드로 빼고, 원문 명사(Entity)를 쌍따옴표 텍스트로 붙여(`...|ETC|3|"종이컵"`)라고 지시하여, 표준(수건)과 비표준(종이컵)이 섞인 복합 요청도 안전하게 분리 추출.
*   **백엔드 방어 로직 (Confidence 강제 하향)**: 분리된 텍스트 중 `ETC` 코드가 있거나 뒤에 임의의 원문이 붙은 결과를 파싱하면, 서버가 이를 비표준 요청으로 인지하고 자체적으로 **해당 태스크의 확신도(Confidence Score)를 하향 조정**(예: 0.9 → 0.6).
*   **관리자 에스컬레이션**: 확신도가 0.7 미만으로 강등되므로, 담당자에게 자동 배정되지 않고 즉시 **시스템 안전망인 관리자(Front) 컨펌 대기열**로 이관(Human-in-the-loop). 관리자가 특이사항("종이컵") 보유 여부를 눈으로 확인 후 최종 승인하거나 반려 처리.

### 4.5 휴먼 피드백 기반 자동 튜닝 (Few-Shot Loop)
*   **전략**: 관리자가 Fallback 된 미분류 태스크의 부서나 코드를 올바르게 "수동 정정"하여 승인하면, 이 양질의 데이터 쌍(원본 ↔ 정답 코드)을 DB에 누적합니다.
*   **활용**: 다음 Gemini API 호출 시 이 데이터를 **Few-shot(퓨샷) 예시 데이터**로 실시간 주입하여, 시스템을 쓸수록 해당 호텔의 도메인(은어나 로컬 외국어) 인식률이 자체 학습(튜닝)되도록 구축합니다.

### 4.6 데이터 플라이휠(Data Flywheel) 기반 RAG 자생적 고도화 파이프라인
*   **콜드 스타트(Cold-Start) 극복 전략**: 오픈 시점에는 최소한의 운영 매뉴얼만 RAG에 입력하여 가볍게 시작합니다.
*   **미답변 데이터 적재 및 클러스터링**: RAG 유사도 기반 검색 실패(정보 없음)로 관리자나 프론트에 에스컬레이션된 고객 질문(예: 주변 맛집, 외부 부대시설 등)만 추려 DB에 별도로 적재합니다. 주기적(예: 주 1회)으로 배치 스케줄러(Cron)가 LLM을 활용해 이 미답변 질문들을 군집화(Clustering)하고 예상 표준 답변의 초안을 생성합니다.
*   **관리자 원클릭 승인(Approve)**: 관리자 대시보드에 노출된 `주간 RAG 업데이트 추천 리스트` 중 적합한 내용만 '최종 승인'하면, 즉시 임베딩(Vector) 과정을 거쳐 RAG 지식 DB(`pgvector`)에 반영됩니다.
*   **기대 효과**: 운영 기간이 길어질수록, 처음 예측하지 못한 고객의 다양한 니즈가 스스로 학습되어 **도메인에 특화된 지식창고로 자생 진화(Evolution)** 하게 됩니다.


### 4.7 AI 기반 교대 인수인계 브리핑 (Shift Handover Automation)
*   **전략 (Scope-down)**: 무거운 외부 스프레드시트 라이브러리 연동 대신, 기존 관리자 대시보드(Data Grid)와 백엔드의 `Request` 엔티티를 활용하여 가볍고 확실한 인수인계 파이프라인을 구축합니다.
*   **데이터 필터링**: 프론트엔드가 별도의 데이터를 취합할 필요 없이, 백엔드에서 해당 교대 시간 내의 주요 태스크 목록을 조회합니다: ① 당일 미처리 태스크 (`status != COMPLETED`) ② 완료되었으나 **관리자 에스컬레이션을 거친 특이사항** (`confidence < 0.7` 또는 `ESCALATED` 이력) ③ 정상 처리된 건 중 **긴급/주요(URGENT/HIGH) 태스크**.
*   **One-pass 요약**: 조회된 태스크 JSON 배열을 그대로 Gemini API에 주입하여, "다음 근무자에게 전달할 미처리 잔여 업무, 오늘 발생했던 특이사항, 그리고 중요도가 높았던 주요 처리 내역을 3~5줄의 자연어로 요약해줘"라고 요청(Prompting)합니다.
*   **기대 효과**: 복잡한 I/O 처리 없이 단 1번의 API 호출만으로 "데이터 기반 자동 브리핑(잔여 업무 + 특이사항 + 주요 처리 내역)"이라는 완성도 높은 하루 단위 업무 루프를 구성합니다. 향후 Phase 2에서 동적 컬럼 매핑 기반의 XLS 양방향 연동으로 확장할 수 있는 기반(Data Source)이 됩니다.

## 5. 인프라 보안 및 네트워크 배포 전략

*   **포트 숨김 및 네트워크 3계층 방어 (논리적 BFF 아키텍처)**:
    *   **Frontend / Proxy (노출)**: 외부 인터넷(모바일 디바이스) 영역과 통신하는 유일한 관문. 브라우저의 CORS 처리 및 보안 쿠키 발급 담당.
    *   **Backend (숨김)**: Spring Boot 코어 서버는 외부 퍼블릭 인터넷(Public IP) 포트를 전면 차단하고, 인프라 내부망(VPC) 안에서 프론트/프록시의 안전한 요청만 수락.
    *   **Database (완전 숨김)**: PostgreSQL 등 DB 포트(5432)는 외부에 절대 개방하지 않음. IoT나 외부 연동 데이터를 수집할 때도 백엔드(Spring)가 Fetch한 뒤 DB로 주입하는 방식 고수. 랜섬웨어 등 외부 공격 표면적(Attack Surface) 제로화.
*   **DB 유지보수를 위한 SSH 터널링**:
    *   DB 포트가 외부와 단절되어 있어도, 관리자나 백엔드 개발자는 외부(자택 등) 환경에서 **SSH 터널링(Secure Shell Tunneling)**을 통해 서버에 뒷문으로 안전하게 연결하여 DBeaver 등의 GUI 툴로 DB 데이터를 100% 원격 가공 및 수정 가능.
*   **무결점 Token 인증 보안 (HttpOnly Cookie)**:
    *   XSS 해킹 공격 방어를 위해, 직원 로그인 인증용 JWT 토큰은 브라우저 공간(Local Storage)에 절대 띄우지 않음. 
    *   자바스크립트가 물리적으로 접근 불가능한 **`HttpOnly`, `Secure` 쿠키** 형태로만 프론트-백엔드 간 교환되도록 강제하여 BFF 토큰 핸들링 보안 패턴의 철학을 단일 아키텍처 환경에서도 그대로 준수.
*   **배포 형태 (Containerized Deployment)**:
    *   모든 WAS 모듈(Core App, WebSocket Node)은 Docker 이미지로 구워내 독립적인 모듈성을 보장. 추후 트래픽 증가 시 K8s로 무중단 스케일 아웃이 가능하도록 론칭.
